## 原文

1. 总体架构
建议做成这 6 层：

定时任务
  ↓
多源采集层：RSS / API / 搜索引擎 / 官网 / 数据库 / 社媒
  ↓
清洗去重层：URL 去重、标题相似度、embedding 聚类、来源归并
  ↓
结构化理解层：提取实体、事件、时间、金额、产品、指标、风险
  ↓
事实验证层：官方源优先、多源交叉验证、置信度评分
  ↓
日报 / 周报 / 看板 / 预警推送
你的目标不是“搜到很多链接”，而是把信息变成：

{
  "事件": "OpenAI 发布新模型",
  "时间": "2026-05-08",
  "涉及主体": ["OpenAI", "ChatGPT"],
  "类型": "AI产品发布",
  "影响": "可能改变Agent开发成本结构",
  "证据": [
    "官方博客",
    "权威媒体",
    "社区讨论"
  ],
  "置信度": 0.92,
  "是否值得关注": true
}
2. 信息源设计：用“分层信源”，不要平均信任所有网页
A. AI 新闻 / 科技发展
类型	推荐来源	用途
官方源	OpenAI、Anthropic、Google DeepMind、Meta AI、Microsoft、NVIDIA、AWS、Google Cloud、Hugging Face Blog	发布、模型、API、产品、价格、能力变化
论文源	arXiv、Papers with Code、ACL、NeurIPS、ICLR、ICML	技术趋势、模型架构、RAG、Agent、新 benchmark
开源源	GitHub API、Hugging Face、GitHub Trending	新项目、star 增长、开源模型、工具链变化
技术社区	Hacker News、Reddit、X、Discord、LessWrong	早期信号、开发者评价、争议
媒体源	The Verge、TechCrunch、Wired、MIT Tech Review、Reuters、Bloomberg、The Information	商业化、融资、公司动向
arXiv 的 API 返回 Atom 格式，适合自动化抓取论文检索结果；GitHub REST API 可用于自动化获取仓库、搜索和开发者生态信号。

B. 币圈新闻 / 链上数据
币圈最容易被噪音污染，所以要把 新闻 和 数据 分开。

类型	推荐来源	用途
市场数据	CoinGecko、CoinMarketCap	价格、市值、交易量、交易所数据
DeFi 数据	DefiLlama	TVL、协议收入、DEX volume、stablecoin supply
链上数据	Dune、Nansen、Arkham、Glassnode、Token Terminal	巨鲸、资金流、协议收入、链上行为
新闻	The Block、CoinDesk、Decrypt、Blockworks、Wu Blockchain	政策、交易所、项目、融资、安全事件
官方源	项目博客、治理论坛、GitHub、X 官方账号	升级、空投、漏洞、治理提案
CoinGecko 官方文档称其 API 覆盖 1,000+ 交易所、18,000+ 币种和 600+ 分类；DefiLlama API 提供 DeFi 协议、TVL、稳定币、DEX、费用和收入等端点。

C. AI 产品横评
AI 产品横评不能只抓官网，因为官网只会说自己好。建议采集：

维度	数据源
产品发布	Product Hunt、官方博客、X、Hacker News
用户评价	Reddit、HN、G2、Capterra、YouTube、Bilibili、小红书、知乎
价格变化	官网 Pricing 页定时快照
功能变化	Changelog、Docs、Release Notes
开发者采用	GitHub、npm、PyPI、Hugging Face
真实能力	自己构建 benchmark 测试集
Product Hunt API 2.0 提供 GraphQL 接口，可用于跟踪新产品发布，但其文档也明确默认不允许商业用途，商业使用需联系 Product Hunt。

D. 电商市场变动
电商不要只看新闻，要重点看 搜索热度、流量、平台报告、价格、销量、广告投放、类目排名。

类型	推荐来源	用途
平台数据	Amazon SP-API、Shopify Admin API、TikTok Shop、Shopee、Lazada	店铺、订单、库存、类目
趋势数据	Google Trends、Similarweb、Semrush、Ahrefs	需求变化、搜索趋势、流量变化
商品价格	Amazon、eBay、AliExpress、Temu、1688、淘宝、京东	价格战、品类波动
广告情报	Meta Ads Library、TikTok Creative Center、Google Ads Transparency	爆品、素材、投放趋势
行业报告	Marketplace Pulse、eMarketer、Statista、艾瑞、QuestMobile	宏观市场
Amazon SP-API 的 Reports API 可用于获取库存、订单、税务、退货等卖家业务报告；ShopifyQL 可通过 Shopify GraphQL Admin API 查询销售、订单等分析数据。

Google 在 2025 年 7 月发布了 Google Trends API alpha，提供最多约 5 年滚动窗口的搜索兴趣数据，并支持日/周/月/年聚合和地区维度，但目前仍是有限测试。

3. 搜索层：不要只用一个搜索 API
建议至少用 三种检索通道：

通道 1：固定源抓取
适合高可信度来源。

RSS / 官网 changelog / 官方博客 / GitHub release / arXiv / CoinGecko / DefiLlama
优点：稳定、便宜、可控。
缺点：只能覆盖你已知的来源。

通道 2：搜索引擎 API
适合发现未知信息。

可选：

工具	适合
Brave Search API	通用 Web、新闻、图片、视频搜索
Tavily	面向 AI Agent / RAG 的搜索和内容抽取
Exa	语义搜索、网页发现、深度研究
SerpAPI	Google 搜索结果结构化抓取
Google Programmable Search	特定域名搜索，但不建议新项目长期依赖
Brave Search API 提供 Web、News、Image、Video 等搜索接口，并强调其独立索引；Tavily 定位于 AI Agent 和 RAG 工作流的实时搜索与内容抽取。

注意：Google Custom Search JSON API 已对新客户关闭，现有客户需要在 2027 年 1 月 1 日前迁移，所以新系统不要重度依赖它。

通道 3：趋势 / 社区 / 异常检测
这是“全面性”的关键。

例如：

GitHub star 暴涨
Product Hunt 排名突然上升
HN 评论量异常
X/Reddit 提及量异常
CoinGecko 交易量异常
DefiLlama TVL 异常
Google Trends 搜索兴趣异常
Similarweb 访问量异常
Amazon 类目排名异常
新闻通常是“已经发生后的解释”，而异常信号常常更早。

4. Agent 分工：建议做成 5 个子 Agent
不建议一个 Agent 干所有事。建议拆成：

1）Search Planner Agent
负责生成搜索计划。

输入：

今天日期、主题、关键词库、实体关注列表、历史热点
输出：

{
  "queries": [
    "OpenAI latest model release May 2026",
    "AI agent framework funding",
    "site:openai.com blog API pricing",
    "新 AI 产品 发布 融资",
    "DeFi TVL change last 24h",
    "Amazon category sales rank changes"
  ],
  "sources": ["official", "media", "github", "producthunt", "crypto_data"],
  "time_range": "last_24h"
}
2）Collector Agent
负责执行搜索和抓取。

能力：

调用 RSS
调用 API
调用搜索引擎
抓取网页正文
提取发布时间
保存原文快照
关键点：原文必须保存，否则后面无法复核。

3）Dedup & Cluster Agent
负责去重和聚类。

去重逻辑：

canonical_url 去重
标题相似度去重
正文 embedding 相似度去重
同一事件多来源聚类
例如：

“OpenAI releases GPT-X”
“OpenAI launches new reasoning model”
“OpenAI 发布新模型”
这三个可能是同一事件，应该合并，而不是日报里写三条。

4）Verifier Agent
负责判断真假和置信度。

建议规则：

证据情况	置信度
官方源 + 多家权威媒体确认	高
官方源单独发布	高
权威媒体独家，暂无官方确认	中高
多个社区讨论，无官方源	中
单一社媒爆料	低
匿名截图 / 搬运号	极低
币圈尤其要加规则：

价格、市值、TVL、交易量必须来自数据 API
项目消息必须找官方公告
安全事件必须找链上交易、审计方、项目方或安全公司
5）Analyst / Report Agent
负责写日报。

不要只总结“发生了什么”，要输出：

发生了什么
为什么重要
影响谁
证据是什么
可信度多高
后续应该追踪什么

7. 准确性机制：必须做“事实级引用”
普通新闻摘要的问题是：模型会把多篇文章混在一起，最后变成“像真的”。

你应该让 Agent 生成这种结构：

{
  "claim": "某公司发布了新 AI 编程工具",
  "evidence": [
    {
      "source": "official_blog",
      "url": "...",
      "quote": "..."
    },
    {
      "source": "tech_media",
      "url": "...",
      "quote": "..."
    }
  ],
  "confidence": "high",
  "needs_human_review": false
}
日报里每个重要判断都要带来源：

结论：这个产品值得关注。
原因：
1. 官方发布了 API；
2. Product Hunt 当日排名靠前；
3. HN 评论中开发者主要讨论成本和准确性；
4. GitHub repo 24 小时 star 增长异常。

10. 最关键的评分系统
建议每条事件打 5 个分：

重要性：这件事影响多大？
新颖性：是不是新消息？
可信度：证据强不强？
行动价值：对你是否有用？
持续追踪价值：后续是否可能发酵？
示例公式：

总分 =
重要性 × 0.30
+ 可信度 × 0.25
+ 新颖性 × 0.15
+ 行动价值 × 0.20
+ 追踪价值 × 0.10
这样你不会被垃圾热点淹没。

11. 我给你的最终建议
最佳路线
你应该分三阶段做：

第一阶段：先做“日报 Agent”

每天自动抓取

自动摘要
每条带来源

第二阶段：做“横评 Agent”

价格
API

适用人群
缺点
是否值得使用
第三阶段：做“情报看板 + 异常预警”
比如：

某 AI 产品突然爆火

某电商品类搜索热度暴涨
某公司发布新模型但社区评价翻车
12. 一句话总结
你要搭的不是“会搜索的 Agent”，而是：

一个以多源数据为基础、以事件聚类为核心、以事实验证为边界、以商业判断为输出的 AI 情报系统。

最小可行方案：

Python + FastAPI + PostgreSQL + Qdrant + RSS + Brave/Tavily + CoinGecko + DefiLlama + arXiv + GitHub + Product Hunt + LLM 结构化总结
最重要的工程原则：

先抓事实，再让模型分析；
先建信源，再做总结；
先做引用，再做观点；
先做去重，再做日报；
先做评分，再做推荐。
